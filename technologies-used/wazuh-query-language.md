# Wazuh Query Language (DQL) Guide

A practical reference for searching and filtering alerts in the Wazuh dashboard —
written against what's actually visible in the `target-ubuntu` agent used throughout
this lab. Wazuh's dashboard is built on **OpenSearch Dashboards** (a Kibana fork), and
its search bar speaks **DQL — the Dashboards Query Language** — the same family of
syntax as Kibana's KQL. This guide covers the syntax itself, the fields that matter for
this lab, and ready-to-use example queries for hunting SSH brute-force activity.

---

## 1. Where to search

Every module in the Wazuh dashboard (**Threat Hunting**, **Configuration Assessment**,
**Vulnerability Detection**, etc.) exposes the same two search surfaces, stacked at the
top of the page:

| Surface | Purpose |
|---|---|
| **Search bar** (`DQL` toggle) | Free-text query using field:value syntax, described below |
| **Filter pills** (`+ Add filter`) | Structured, clickable filters (`field` / `operator` / `value`) — equivalent to DQL but built through a UI form instead of typed |
| **Time picker** (top right, e.g. "Last 24 hours") | Restricts the query to a time window — always applied *in addition to* whatever is in the search bar |

Filter pills and the DQL search bar are two views of the same underlying query — a pill
like `agent.id: 006` is exactly equivalent to typing `agent.id: 006` into the search bar.
Screenshots throughout this lab (e.g. `hydra-lab/images/04-wazuh-dashboard-alert-summary.png`)
show both in use simultaneously: pinned filter pills for `manager.name` and `agent.id`,
with the free-text bar left empty for additional ad-hoc queries.

## 2. DQL syntax reference

### 2.1 Basic field match

```
field: value
```

```
agent.name: target-ubuntu
rule.level: 5
```

Values with spaces must be quoted:

```
rule.description: "sshd: authentication failed"
```

### 2.2 Wildcards

`*` matches any number of characters within a value:

```
rule.description: *authentication*
data.srcip: 172.18.*
```

### 2.3 Boolean operators

`and`, `or`, `not` (case-insensitive), with parentheses for grouping:

```
rule.groups: sshd and rule.level >= 5
rule.id: 5760 or rule.id: 5503
not rule.groups: sca
(rule.groups: sshd or rule.groups: pam) and agent.id: 006
```

### 2.4 Ranges

Numeric and date fields support comparison operators:

```
rule.level >= 5
rule.level > 3 and rule.level < 10
timestamp >= "2026-07-19T13:37:00" and timestamp <= "2026-07-19T13:38:00"
```

### 2.5 Existence checks

Match documents where a field is present at all:

```
data.srcip: *
```

### 2.6 Nested / dotted fields

Wazuh alerts are deeply nested JSON; reference nested fields with dot notation exactly
as they appear in the **Expand** view of any alert row:

```
data.sca.check.result: failed
data.win.eventdata.logonType: 10
```

### 2.7 What DQL is *not*

DQL is not full Lucene syntax — there's no `+`/`-` prefix syntax, no fuzzy (`~`) search,
and no proximity search. For that level of control, switch the query-language toggle
from `DQL` to `Lucene` in the search bar (available in most Wazuh dashboard versions),
which accepts classic Lucene query syntax (`field:value~2`, `field:[1 TO 5]`, etc.).

## 3. Fields that matter for this lab

These are the fields actually populated on the alerts generated by the Hydra attack and
visible in the event tables (`hydra-lab/images/05-wazuh-alert-events-detail.png`):

| Field | Example value | Meaning |
|---|---|---|
| `agent.id` | `006` | The Wazuh agent's numeric ID (target-ubuntu) |
| `agent.name` | `target-ubuntu` | The agent's registered hostname |
| `manager.name` | `wazuh.manager` | Which Wazuh manager processed the event (relevant in multi-manager clusters) |
| `rule.id` | `5760` | The specific rule that fired — see [`technologies-used/wazuh.md`](technologies-used/wazuh.md) for the full ID reference used in this lab |
| `rule.level` | `5` | Alert severity, 0 (informational) to 16 (critical) |
| `rule.description` | `sshd: authentication failed` | Human-readable description from the rule definition |
| `rule.groups` | `["syslog","sshd","authentication_failed"]` | Array of tags a rule belongs to — the single most useful field for broad hunting |
| `rule.mitre.id` | `T1110.001` | MITRE ATT&CK technique ID mapped to the rule |
| `rule.mitre.tactic` | `Credential Access` | MITRE tactic mapped to the rule |
| `data.srcip` | `172.18.0.x` | Source IP of the connection (populated for `sshd` rules via the decoder) |
| `data.srcuser` | `victim` | Username the login was attempted against |
| `data.sca.check.result` | `passed` / `failed` | SCA (Configuration Assessment) check outcome |
| `data.sca.policy` | `CIS Ubuntu Linux 22.04 LTS Benchmark v1.0.0.` | Which compliance policy the SCA check belongs to |
| `full_log` | *(raw log line)* | The original, undecoded log text — always available as a fallback search target |
| `timestamp` | `2026-07-19T13:37:45.237Z` | When the manager processed the event |

> **Tip:** click the magnifying-glass icon on any row in the Events table (visible on the
> left of every row in `05-wazuh-alert-events-detail.png`) to expand the full JSON
> document and see every field available for that specific alert — the fastest way to
> discover a field name without guessing.

## 4. Practical example queries

All examples below assume the `agent.id: 006` (`target-ubuntu`) filter pill is already
applied, as it was throughout this lab, and a time range covering the attack window.

### 4.1 All SSH-related alerts

```
rule.groups: sshd
```
Broadest possible SSH hunting query — catches every rule tagged `sshd`, regardless of
outcome (failed, success, invalid user, protocol errors, scanning, etc.).

### 4.2 Only failed SSH authentication attempts

```
rule.id: 5760
```
The exact rule this lab's Hydra failures triggered. Narrower and more precise than
`rule.groups: authentication_failed`, which also includes PAM-side failures (`5503`).

### 4.3 Both sshd- and PAM-side failure alerts together

```
rule.id: 5760 or rule.id: 5503
```
Reconstructs the *complete* failure picture for a login attempt — remember each Hydra
guess produced **two** correlated alerts (one from `sshd`'s own log line, one from PAM's
independent log line), so this query is how you'd count total failure *events*, not
total failed *attempts* (divide by ~2 for the latter).

### 4.4 Successful logins only

```
rule.id: 5715
```
In this lab, exactly one hit — the moment Hydra's `password123` guess succeeded. In
production, this is the query you'd pair with an "unexpected successful login" watch.

### 4.5 High-severity events, regardless of source

```
rule.level >= 10
```
Level 10 is where Wazuh's built-in brute-force *correlation* rules live (`5551` PAM,
`5712`/`5720` sshd — see [`technologies-used/wazuh.md`](technologies-used/wazuh.md)).
As noted in the [incident report](incident-response/hydra-lab-1-report.md), this lab's
8-attempt wordlist stayed *under* that threshold, so this query returns nothing for the
attack window — itself a useful negative-result check.

### 4.6 Everything mapped to Brute Force (MITRE T1110)

```
rule.mitre.id: T1110*
```
Technique-centric hunting: find every alert mapped to brute force, regardless of which
specific rule (`5710`, `5712`, `5716`, `5760`, `5503`, `5551`, `5720` all map here) or
protocol fired it. The wildcard also matches sub-techniques like `T1110.001`.

### 4.7 Full brute-force reconstruction for one username

```
data.srcuser: victim and (rule.groups: sshd or rule.groups: pam)
```
Pulls every SSH- or PAM-tagged event for a specific target account — the query you'd run
to build the exact timeline in [§5 of the incident report](incident-response/hydra-lab-1-report.md)
from scratch.

### 4.8 Narrow time window around the attack burst

Using the time picker (not DQL) set to a custom range:

```
Jul 19, 2026 @ 13:37:40.000 → Jul 19, 2026 @ 13:37:48.000
```
combined with:
```
agent.id: 006
```
This is how the ~6.5-second attack burst was isolated from the noisier 24-hour default
window — always narrow the time range *before* trying to narrow further with query
syntax; it's cheaper for the indexer and easier to read.

### 4.9 Configuration Assessment failures for the target

```
rule.groups: sca and data.sca.check.result: failed
```
Independent of the attack — finds every CIS benchmark control the target host is
currently failing, which is what produced the 42% compliance score referenced in
[`hydra-lab-1-report.md §4`](incident-response/hydra-lab-1-report.md#4-detection).

### 4.10 Exclude noise, keep only actionable auth events

```
(rule.groups: authentication_failed or rule.groups: authentication_success) and not rule.groups: sca
```
A general-purpose "just show me login activity" query — useful as a saved search when
you want the dashboard to default to signal over the startup/housekeeping noise (`ossec`,
`rootcheck`, `syslog` grouping alerts) that fills most of a 24-hour window in a small lab.

## 5. Building a saved search / visualization

Any query above can be turned into a reusable asset:

1. Run the query with the correct time range.
2. Click the **save** icon (floppy-disk icon, left of the search bar — visible in
   `hydra-lab/images/05-wazuh-alert-events-detail.png`) to save it as a **Saved Search**.
3. Saved searches can be added directly to a custom **Dashboard** panel, or used as the
   data source for a new **Visualization** (e.g. a bar chart of `rule.id` counts over
   time) via **Menu → Dashboards Management → Saved Objects**.

This is how the built-in panels seen throughout this lab (Top 5 alerts, Top 10 Alert
groups evolution, MITRE ATT&CK Top Tactics) are constructed under the hood — they are
all just DQL queries + an aggregation, rendered as a chart.
