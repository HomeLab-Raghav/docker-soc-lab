# 05 — Volumes and Persistence

## The problem: containers are ephemeral

A container's filesystem is temporary. When a container is removed, everything written inside it disappears. That's actually a feature — containers are meant to be disposable. But for a SIEM, you don't want your logs and alert history to vanish every time you restart.

Docker solves this with **volumes**: storage that lives outside the container's lifecycle.

---

## Two types of volume mounts in this lab

### Named volumes

Docker manages where the data actually lives on your host. You just give it a name:

```yaml
# In docker-compose.yml
volumes:
  wazuh_logs:/var/ossec/logs
  wazuh_etc:/var/ossec/etc
  wazuh-indexer-data:/var/lib/wazuh-indexer
```

And declare them at the bottom of the file:

```yaml
volumes:
  wazuh_api_configuration:
  wazuh_etc:
  wazuh_logs:
  wazuh_queue:
  wazuh_var_multigroups:
  wazuh_integrations:
  wazuh_active_response:
  wazuh_agentless:
  wazuh_wodles:
  filebeat_etc:
  filebeat_var:
  wazuh-indexer-data:
  wazuh-dashboard-config:
  wazuh-dashboard-custom:
```

That's 14 named volumes in this lab. When you run `docker-compose up` for the first time, Compose creates all of them. When you run `docker-compose down`, the containers stop but the volumes stay. When you run `docker-compose up -d` again, the containers pick up right where they left off.

### Bind mounts

You point directly to a file or folder on the host:

```yaml
# In docker-compose.yml — mounting SSL certs from ./config/ into containers
- ./config/wazuh_indexer_ssl_certs/root-ca-manager.pem:/etc/ssl/root-ca.pem
- ./config/wazuh_cluster/wazuh_manager.conf:/wazuh-config-mount/etc/ossec.conf
```

The `./` path is relative to the directory where `docker-compose.yml` lives. This is how the SSL certificates generated by the cert generator get into the containers: the generator writes them to `./config/wazuh_indexer_ssl_certs/` on the host, and each service mounts the relevant cert from there.

---

## What each volume holds

| Volume | Container path | What's in there |
|---|---|---|
| `wazuh_etc` | `/var/ossec/etc` | Wazuh config, `client.keys` (agent enrollment records) |
| `wazuh_logs` | `/var/ossec/logs` | Manager logs, `alerts/alerts.json` |
| `wazuh_queue` | `/var/ossec/queue` | Wazuh message queue |
| `wazuh-indexer-data` | `/var/lib/wazuh-indexer` | All OpenSearch indices (your alert history) |
| `wazuh-dashboard-config` | `...data/wazuh/config` | Dashboard settings |
| `filebeat_etc` | `/etc/filebeat` | Filebeat config (how manager ships to indexer) |

The most important one for the lab is `wazuh-indexer-data` — it holds every alert that's ever been indexed. Losing it means losing your alert history.

---

## `docker-compose down` vs `docker-compose down -v`

```powershell
# Stop containers, keep volumes (data survives)
docker-compose down

# Stop containers AND delete all named volumes (data wiped)
docker-compose down -v
```

Use `down` when you're done for the day. Use `down -v` only when you want a completely fresh start — after `down -v` you'll need to re-run the cert generator and all your alert history is gone.

---

## Why target-ubuntu has no named volumes

`target-ubuntu` uses a plain `ubuntu:22.04` image with no volumes defined. Its data — the installed packages, the Wazuh agent, the victim user — live in the container's writable layer. That means:

- `docker-compose -f agent.yml down` + `up -d` reinstalls everything from scratch (slow, ~60s)
- `docker restart target-ubuntu` reuses the existing writable layer (fast, ~5s)

This is fine for a disposable attack target. If you wanted the agent install to persist across `down`/`up` cycles, you'd add a named volume for `/var/ossec`.

## Inspecting volumes

```powershell
# List all volumes on the host
docker volume ls

# See details about one volume (including where Docker stores it)
docker volume inspect docker-soc-lab_wazuh_logs

# See how much disk space volumes are using
docker system df -v
```

On Windows, Docker stores named volume data inside the WSL2 VM. You can't browse it in Windows Explorer, but you can reach it from WSL:

```bash
# From WSL2
ls /var/lib/docker/volumes/docker-soc-lab_wazuh_logs/_data
```
